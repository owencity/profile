# 24시간이 모자라 - 문서

이 폴더(`docs/`)는 **프로젝트의 설계/운영/개발 규칙**을 누적 관리합니다.  
루트 `README.md`는 “실행/요약/링크” 중심, 이 문서는 “상세” 중심으로 유지합니다.

## 프로젝트 한 줄 소개

**24시간 운영하는 카페/음식점/전기차 충전소를 조회**하고, 결과를 **지도 위에 표시**하며, **내 주변 1km 반경 검색**을 지원하는 웹앱.

## 스택

- **Frontend**: React + TypeScript + Vite + Tailwind CSS
- **Backend**: Java (별도 프로젝트, AWS EC2 배포)
- **Database**: PostgreSQL

## 요구사항(상세)

1. **장소 조회**
   - 카페, 음식점, 전기차 충전소 중 **24시간 운영**하는 곳을 조회할 수 있어야 함
2. **지도 표시**
   - 조회된 카페/음식점/충전소를 **지도 위에 마커로 표시**해야 함
3. **내 주변 1km 반경 조회**
   - 사용자의 현재 위치 기준 **1km 반경** 필터링/조회가 가능해야 함
4. **테스트 코드**
   - 기능 구현 시 테스트 코드를 작성하되, **필요할 때만** 작성
5. **기능 단위 진행 원칙**
   - 기능 하나를 만들 때마다 **설계 → 구현 → 완료 → 코드리뷰 → 개선점 반영** 흐름을 강제
   - “무의식적인 진행” 금지 (작업 전에 산출물을 남기고 진행)

## 현재 상황

- **웹 초기 UI/UX 완성(목업)**
- **데이터 크롤링 완료**
  - 전기차
  - 24시간 카페
- **다음 스텝 준비 중**

## 역할/진행 방식(팀 규칙)

- **백엔드**: 팀장님이 직접 진행
  - 단, AI는 “도구”로만 사용(자동 진행 X)
- **설계/아키텍처**: 팀장님이 결정
  - AI는 **조언/리스크 체크/대안 제시** 역할
- **그 외 작업(프론트 구현/문서/정리 등)**: AI가 전담

## 기능 개발 프로세스(권장 템플릿)

기능 하나를 시작할 때 아래를 최소 산출물로 남깁니다.

### 1) 설계(필수)

- **기능 목표**: 무엇을, 어떤 입력으로, 어떤 출력으로 제공하는가
- **유저 시나리오**: 사용자가 어떤 흐름으로 쓰는가
- **UI/상태 설계**: 화면 구성 / 로딩 / 에러 / 빈 상태
- **API 계약(프론트 관점)**:
  - 엔드포인트, 요청/응답 형태, 에러 포맷
  - 인증/권한, 페이징/정렬/필터
- **지도 표현 규칙**:
  - 마커/클러스터링 여부, 카테고리별 스타일, 선택 상태
- **성능/비용 고려**:
  - 요청 횟수 제한, 디바운스, 캐싱, 범위 검색 전략

### 2) 구현(필수)

- 작은 단위로 커밋/PR을 쪼개고, 작업 범위를 문서에 명확히 기록

### 3) 테스트(선택)

**필요할 때만 작성**하되, 아래 중 하나라도 해당하면 작성 권장:

- 거리 계산(1km 반경) 등 **로직 실수 비용이 큰 경우**
- 지도 마커/필터가 복잡해져 **회귀 가능성이 높은 경우**
- 크롤링 데이터/백엔드 응답이 다양해 **엣지 케이스가 많은 경우**

### 4) 코드리뷰 & 개선(필수)

- 완료 기준(DoD)을 만족하는지 체크
- 성능/UX/에러 핸들링/코드 품질 개선점을 반드시 1개 이상 도출

## 다음 스텝(제안)

아래는 일반적인 다음 단계 후보입니다. 팀장님 아키텍처 결정에 맞춰 조정합니다.

- **API 연동 방식 확정**: `VITE_API_BASE_URL` 기반으로 환경 분리 
- **지도 SDK 확정/적용**: 마커/필터/반경 표시 UX 결정
- **내 주변 1km 반경 로직**: 좌표 기반 필터(프론트/백 중 어디서 처리할지 결정)
- **카테고리 필터**: 카페/음식점/충전소 토글 및 상태 유지
- **앱 배포(24hours 전용)**: `VITE_APP_TARGET=24hours`로 앱 전용 빌드 분리
